# Contributing

## Workflow

- Keep pull requests small and scoped to one concern.
- Prefer a single clear implementation over multiple speculative options.
- Add or update verification when behavior changes.
- Remove obsolete code instead of preserving duplicate paths.

## Review expectations

A change is ready when it answers these questions:

- What changed?
- Why is this the right change?
- How was it verified?
- What, if anything, remains intentionally out of scope?

## Commit hygiene

- Use direct, descriptive commit messages.
- Avoid noisy formatting-only commits unless formatting is the actual change.
- Do not mix unrelated refactors and behavior changes.

## Documentation

If a change affects usage, operating assumptions, or safety boundaries, update the relevant docs in the same pull request.
